# carve-harness — 정량 평가 벤치마크 체계

> **목적**: 완성된 carve-harness가 다른 하네스(베이스라인: 무하네스 단일 에이전트 / OpenHarness / ECC / Squad-only)보다 우수함을 **재현 가능한 수치**로 증명한다.
>
> | 항목 | 값 |
> |------|----|
> | 문서 버전 | v1.1 |
> | 작성일 | 2026-05-31 |
> | 측정 철학 | "계산·측정 먼저, 주장은 나중" — 모든 수치는 재현 명령과 함께 |

---

## 결론 (먼저)

하네스 우수성은 **6개 축**으로 측정한다. 핵심은 *"같은 작업·같은 모델·같은 프로젝트에서 하네스만 바꿔" 비교*하는 것(통제 변인 고정).

| # | 평가 축 | 한 줄 질문 | 핵심 지표 |
|:-:|---------|-----------|----------|
| 1 | **속도/효율** | 같은 작업을 더 적은 토큰·시간·비용으로? | 토큰/태스크, 벽시계 시간, $/태스크, KV-cache hit |
| 2 | **제어/안전** | 위험 행동을 결정적으로 막는가? | 차단 정확도, 권한 위반 누출률, 오차단율(FP) |
| 3 | **프롬프트 검증** | 생성된 프롬프트가 의도대로 트리거·동작하는가? | 트리거 정확도, 지시 이행률 |
| 4 | **컨텍스트 검증** | 컨텍스트를 효율·정확하게 관리하는가? | 컨텍스트 점유율, 압축 후 정보 보존율, 40% 룰 준수 |
| 5 | **기능 동작 (E2E)** | 스킬·훅·에이전트가 실제로 도는가? | 스킬 발화율, 훅 발동률, E2E 통과율 |
| 6 | **구성 품질** | 자동 구성된 하네스 자체가 좋은가? | 구성 정확도, 멱등성, audit 청결도 |

> **차별화 가설**: carve-harness의 우위는 "최고 성능"이 아니라 **"프로젝트에 맞게 자동으로 군더더기를 깎아낸 효율"**에 있다. 따라서 #1(효율)과 #6(구성 정확도)이 핵심 차별 축이고, #2~#5는 "깎아도 품질이 안 떨어진다"는 안전성 증명이다.

## 0. 측정 프로토콜 (공정성 규칙)

벤치마크가 신뢰받으려면 비교 조건을 고정해야 한다.

```
통제 변인 (고정):  동일 모델 · 동일 태스크셋 · 동일 프로젝트 스냅샷(git SHA) · 동일 시드
독립 변인 (변경):  하네스 종류만 교체
   A. No-Harness  (단일 에이전트, 베이스라인)
   B. OpenHarness (oh)
   C. ECC         (everything-claude-code)
   D. Squad-only  (subagents)
   E. carve-harness (대상)
반복:  각 조건 최소 5회 (LLM 비결정성 → 중앙값·IQR 보고, 평균 단독 금지)
```

- **태스크셋**: 난이도별 3종 — ① 단순 CRUD ② 멀티기능 앱 ③ 리팩토링/버그픽스. 각 5~10개 태스크
- **보고 형식**: 단일 숫자 금지. `중앙값 (IQR), n=5` 형태. 베이스라인 대비 **배수/델타**로 표기
- **재현성**: 모든 측정은 스크립트 1개로 실행 가능해야 함 (`bench/run.sh`)

## 1. 속도 / 효율 (Efficiency)

| 지표 | 정의 | 측정 방법 | 목표(vs No-Harness) |
|------|------|----------|---------------------|
| 토큰/태스크 | 태스크 완료까지 총 input+output 토큰 | `ccusage` 세션 합계 | 동등 작업서 ↓ |
| 벽시계 시간 | 트리거~완료 실시간 | `time` / 타임스탬프 로그 | 단순 작업 ↓ |
| $/태스크 | API 비용 | ccusage 비용 합계 | ↓ (불필요 에이전트 제거 효과) |
| KV-cache hit rate | 캐시 적중 토큰 비율 | API 응답의 cache_read 토큰 / 총 input | ↑ (Stable Prefix 설계 시) |
| 컨텍스트 주입 비용 | 세션 시작 시 구조 파악 토큰 | `/context` 시작값 | ↓ (codesight 연동 시 6~13배) |

**측정 방법·활용 예시:**
```bash
# 토큰·비용 — ccusage로 세션 단위 측정
npx ccusage@latest                    # 세션별 토큰·$ 집계
# 동일 태스크를 A~E 하네스로 각각 5회 → 중앙값 비교표 생성

# 컨텍스트 주입 비용 — Claude Code 내에서
/context                              # 세션 시작 직후 점유 토큰 기록
```
> **활용**: "carve-harness는 단순 CRUD에서 ECC 대비 토큰 X배 절감 (불필요한 119개 스킬 미로딩, 프로젝트 필요분만 깎아 구성)" 같은 주장을 이 표로 뒷받침한다. **비율로 말하되 절대 수치도 병기**한다.

> **주의 (할루시네이션 방지)**: "X배 빠르다"는 반드시 측정 후 기재. 추정계수(1토큰≈4chars)는 보수적 하한으로 명시하고, 비율은 추정계수와 무관함을 밝힌다.

## 2. 제어 / 안전 (Control & Safety)

하네스의 본질인 "제약"이 결정적으로 작동하는지 측정한다.

| 지표 | 정의 | 측정 방법 | 목표 |
|------|------|----------|------|
| 차단 정확도 | 위험 행동 시도 중 실제 차단 비율 | 위험 커맨드 시드셋 N개 주입 → 차단 카운트 | 100% (결정적) |
| 권한 위반 누출률 | flight-rule 위반이 통과된 비율 | 위반 시드셋 → 통과 카운트 | 0% |
| 오차단율 (FP) | 정상 행동을 잘못 막은 비율 | 정상 시드셋 → 차단 카운트 | 최소화 |
| 차단 결정성 | 동일 입력 반복 시 동일 차단 | 같은 위반 5회 반복 | 100% 일관 |

**측정 방법:**
```bash
# 위험 커맨드 시드셋 (예: rm -rf, DROP TABLE, secret 하드코딩)을
# 각 하네스 환경에서 시도 → PreToolUse 훅 exit code 2 발생 여부 카운트
# 결정적 훅(코드 기반)은 100% 기대, 프롬프트 기반 제약은 누출 발생 → 이 차이가 우위
```
> **차별 포인트**: CLAUDE.md(권고)만 쓰는 하네스 vs carve-harness의 **결정적 훅(강제)**. "exit code 2가 유일한 진짜 차단 수단"임을 누출률 0% vs N%로 증명한다.

## 3. 프롬프트 검증 (Prompt / Agent Triggering)

생성된 스킬·에이전트·커맨드가 의도대로 발화·동작하는지.

| 지표 | 정의 | 측정 방법 | 목표 |
|------|------|----------|------|
| 트리거 정확도 | 의도한 상황에서 스킬/에이전트가 발화한 비율 | 트리거 프롬프트셋 → 발화 로그 | ↑ |
| 오발화율 | 무관한 상황에서 잘못 발화 | 비트리거 프롬프트셋 | ↓ |
| 라우팅 정확도 | 올바른 에이전트로 위임된 비율 | Squad-router 키워드 테스트 방식 | ↑ |
| 지시 이행률 | 시스템 프롬프트 지시를 따른 비율 | 체크리스트 항목별 PASS | ↑ |

**측정 방법 (Squad의 테스트 패턴 차용):**
```bash
# subagents의 test-router.sh(67 케이스: skip/conflict/keyword/priority) 방식
# → carve가 생성한 라우팅 규칙에 키워드 시드셋 입력 → 기대 에이전트와 일치율 측정
bash bench/test-trigger.sh        # 트리거 정확도/오발화 카운트
```
> **활용**: "carve가 프로젝트 컨벤션을 주입해 생성한 evaluator 스킬은 트리거 정확도 X%, 범용 스킬은 Y%" — 깎기(carve)의 효과를 발화 정확도로 입증.

## 4. 컨텍스트 검증 (Context Management)

| 지표 | 정의 | 측정 방법 | 목표 |
|------|------|----------|------|
| 컨텍스트 점유율 | 작업 중 컨텍스트 사용 % | `/context` 주기 측정 | 40% 룰 내 유지 |
| 압축 후 보존율 | /compact 후 핵심 정보 유지 비율 | 압축 전후 핵심 항목 체크 | ↑ |
| 조기 완료 선언율 | 미완성을 완료로 보고한 비율 | Context Anxiety 측정 | ↓ |
| on-demand 로딩 효율 | 필요 시점에만 스킬 로딩된 비율 | 스킬 로드 로그 vs 사용 | ↑ (Progressive Disclosure) |

**측정 방법:**
```bash
# 긴 작업(태스크셋 ②) 실행 중 /context를 N턴마다 기록
# → 점유율 곡선이 40%를 언제 넘는지, 압축 후에도 작업 일관성 유지되는지 비교
```
> **근거**: 40% 룰(HumanLayer 12-Factor) — 컨텍스트 40% 초과 시 지시 이행 급락. carve가 불필요 자산을 깎아 점유율을 낮게 유지하면 이 지표에서 우위.

## 5. 기능 동작 — 전체 E2E (Functional)

| 지표 | 정의 | 측정 방법 | 목표 |
|------|------|----------|------|
| 스킬 발화율 | 생성 스킬이 실제 호출된 비율 | 세션 로그 분석 | ↑ |
| 훅 발동률 | 생성 훅이 이벤트에 실제 발동 | settings.json 훅 로그 | 100% |
| E2E 통과율 | Sprint Contract 완료 조건 PASS 비율 | Evaluator 판정 / Playwright | ↑ |
| 태스크 성공률 | 태스크가 실제 동작하는 결과물로 완료 | 독립 검증 | ↑ |
| 회귀 안전성 | 변경이 기존 기능을 깨지 않은 비율 | 테스트 스위트 통과 | ↑ |

**측정 방법:**
```bash
# Evaluator에 Playwright MCP 제공 → 실행 중인 앱을 직접 조작하며 완료 조건 검증
# 각 하네스로 동일 태스크셋 빌드 → E2E PASS/FAIL 카운트
# "단일 $9(broken) vs 풀 하네스 $200(동작)" 같은 품질-비용 곡선 작성
```
> **활용**: 품질(E2E 통과율) × 비용($) 산점도로 "carve는 ECC 품질에 근접하면서 비용은 Squad 수준"이라는 파레토 우위를 시각화.

## 6. 구성 품질 (Auto-Configuration Quality) — carve 고유 축

다른 하네스엔 없는, **자동 구성** 자체의 품질. carve의 핵심 차별 축.

| 지표 | 정의 | 측정 방법 | 목표 |
|------|------|----------|------|
| 구성 정확도 | 프로젝트에 맞는 자산을 정확히 생성한 비율 | 정답 라벨셋 대비 precision/recall | ↑ |
| 과생성률 | 불필요 자산을 생성한 비율 | 생성 자산 중 미사용 비율 | ↓ (깎기 효과) |
| 누락률 | 필요한 자산을 빠뜨린 비율 | 정답 대비 누락 | ↓ |
| 멱등성 | 재실행 시 사용자 자산 보존율 | 수정 후 재실행 → diff | 100% 보존 |
| audit 청결도 | 생성물 보안 위반 건수 | auditor/AgentShield 스캔 | 0건 |
| 구성 시간 | 트리거~완성 시간 | 타임스탬프 | 합리적 범위 |

**측정 방법:**
```bash
# 라벨셋: 스택별로 "있어야 할 자산"을 사람이 정답 정의 (예: Next.js+TS → evaluator + any금지 룰 + 린트훅)
# carve 실행 → 생성물과 정답 비교 → precision/recall 산출
npx ecc-agentshield scan          # 생성물 보안 스캔 (secret/권한/hook injection/MCP/config)
carve install && carve install    # 2회 실행 → 멱등성 diff 확인
```
> **차별 포인트**: 이 축은 OpenHarness·ECC·Squad엔 측정 대상 자체가 없다(수동 구성이므로). carve만이 "자동 구성 정확도"를 가지며, 여기가 경쟁 하네스가 따라올 수 없는 영역이다.

## 8. 재현 가능한 벤치마크 하니스 (구현 권장)

```
carve-harness/
└── bench/
    ├── run.sh                 # 전체 벤치 1커맨드 실행
    ├── tasksets/              # 난이도별 태스크셋 (CRUD/멀티/리팩토링)
    ├── harnesses/             # A~E 환경 셋업 스크립트
    ├── seeds/                 # 위험/위반/트리거 시드셋
    ├── labels/                # 구성 정확도 정답 라벨
    ├── collect.js             # ccusage·/context·로그 수집
    └── report.js              # 스코어카드 자동 생성 (중앙값·IQR)
```

> **원칙 (강의 벤치마크 작성 규칙 준수)**: 수치는 반드시 직접 측정 후 기재. 모든 시나리오는 위 스크립트로 누구나 재현 가능해야 하며, 2차 출처(제작사 공개 수치)는 명시적으로 라벨링한다.
