# 7.1.4 SubAgent CLI의 기능과 구현 I

## 하네스 엔지니어링 — 서브에이전트 시스템 설계

> 기준 프로젝트: `claude-code-expert/subagents` (Squad Agent v1.3.2)
> 8개의 전문 에이전트 + 9개 슬래시 커맨드 + 3개 훅으로 구성된 Claude Code 서브에이전트 시스템

---

## 학습 목표

이 장에서 다루는 내용

1. **서브에이전트가 왜 필요한가** — 컨텍스트 격리·도구 제한·모델 라우팅의 가치
2. **에이전트 정의 형식** — frontmatter + system prompt 구조
3. **8개 에이전트의 역할 분담** — 단일 책임 원칙(SRP)을 에이전트 레벨에 적용
4. **모델 라우팅 전략** — opus / sonnet / haiku를 작업 성격에 맞춰 분배
5. **도구 권한 모델** — Read-only / Write / Bash 도구를 하드 제약으로 강제

## 1. 서브에이전트란?

### 정의

서브에이전트(sub-agent)는 메인 Claude Code 세션 안에서 **독립된 컨텍스트 윈도우**를 갖고 동작하는 전문화된 AI 인스턴스다.

### 일반 호출 vs 서브에이전트 호출

```
[일반 호출 — 인라인]
사용자 → Claude (메인 컨텍스트)
        └─ git diff (30k) + 파일 읽기 (16k) + 분석 (15k) 모두 메인에 누적

[서브에이전트 호출]
사용자 → Claude (메인 컨텍스트)
        └─ Agent(subagent_type="squad-review")
            └─ 새 컨텍스트 윈도우에서 git diff·파일·분석 실행
            └─ 최종 요약(2k)만 메인 컨텍스트로 반환
```

---

## 2. 내부 동작 메커니즘

### Claude Code의 Agent 도구를 통한 호출

```
1. 사용자: "/squad-review src/auth/"

2. 메인 세션 → Agent(subagent_type="squad-review", prompt="...")

3. 새 컨텍스트 윈도우 생성:
   - squad-review.md의 시스템 프롬프트 로드
   - frontmatter에 명시된 도구만 사용 가능
   - frontmatter에 명시된 모델 사용

4. 서브에이전트가 자체 컨텍스트에서 작업:
   - git diff, 파일 읽기, 분석 — 모두 서브에이전트 컨텍스트에만 남음

5. 결과 반환:
   - 최종 메시지만 메인 세션에 반환
   - 서브에이전트 컨텍스트는 폐기
```

> ⚠️ `bash claude -p`로 외부 프로세스를 띄우는 방식이 **아니다**. 동일 프로세스 내부에서 Agent 도구로 위임된다.

---

## 3. 왜 쓰는가 — 4가지 가치

### ① 컨텍스트 격리 — 메인 오염 방지
30k 크기의 git diff가 서브에이전트 컨텍스트에만 남고, 메인에는 2k 요약만 반환된다. 메인 컨텍스트가 작을수록 매 턴의 input token 비용이 줄어든다.

### ② 도구 제한 — 프롬프트가 아닌 하드 제약
`squad-review`는 Read-only로 설계되어 코드를 절대 수정할 수 없다. "수정하지 말라"는 프롬프트 지시가 아니라, **Write 도구가 아예 없다**는 도구 레벨의 제약이다.

### ③ 병렬 실행 — 독립 분석 동시 처리
여러 모듈을 동시에 분석할 때 N개의 서브에이전트를 병렬로 띄울 수 있다.

### ④ 모델 라우팅 — 작업별 비용 최적화
보안 감사는 opus, 커밋 메시지 작성은 haiku로 분배하여 정확도와 비용을 동시에 잡는다.

## 4. 장기 토큰 절감 효과

> "단일 호출 비용은 더 비싸지만, 세션이 길어질수록 누적 절감이 기하급수적이다."

```
[20개 파일 리뷰 — 단일 호출 기준]
인라인:        85k 메인 컨텍스트
서브에이전트:  26k 메인 + 69k 서브(폐기) = 총 95k (단발은 +12% 비쌈)
```

```
[10턴 누적 효과]
메인 컨텍스트는 매 턴 input token으로 재전송됨
인라인:        85k × 10턴 = 850k 누적
서브에이전트:  26k × 10턴 = 260k 누적
차이:          590k 토큰 절감
```

**결론:** 서브에이전트는 **컨텍스트 위생(context hygiene) 도구**다. 짧은 세션엔 손해, 긴 세션엔 압도적 이득.

## 5. 에이전트 정의 형식

### 파일 구조 — `~/.claude/agents/squad-{name}.md`

```markdown
---
name: squad-review                 # 에이전트 ID (Agent 도구의 subagent_type)
description: >                     # 자동 위임 트리거 — Claude가 이 텍스트로 판단
  Use PROACTIVELY after code changes.
  Pipeline: APPROVE → /squad-qa, REQUEST_CHANGES → /squad-refactor
tools: Read, Bash, Glob, Grep      # 허용 도구 (하드 제약)
model: opus                        # haiku / sonnet / opus / inherit
maxTurns: 15                       # 무한 루프 방지
---

당신은 시니어 스태프 엔지니어로서 코드 리뷰를 수행한다.
[시스템 프롬프트 — 페르소나, 절차, 출력 형식]
```

> 두 부분으로 구성: **frontmatter (메타데이터·도구·모델)** + **시스템 프롬프트 (행동 규약)**

## 6. Frontmatter 필드 레퍼런스

| 필드 | 필수 | 기본값 | 설명 |
|------|------|--------|------|
| `name` | ✅ | — | 에이전트 ID. `Agent(subagent_type="...")`에 매칭 |
| `description` | ✅ | — | 자동 위임 트리거. `PROACTIVELY` 포함 시 자동 호출 |
| `tools` | ❌ | (상속) | 콤마 구분 도구 목록. `Task(squad-name)`로 에이전트간 호출 |
| `model` | ❌ | inherit | `haiku` / `sonnet` / `opus` / `inherit` |
| `maxTurns` | ❌ | — | 최대 턴 제한 (무한 루프 방지) |
| `permissionMode` | ❌ | — | `plan` / `acceptEdits` / `bypassPermissions` |
| `memory` | ❌ | — | `user` / `project` / `local` |
| `background` | ❌ | false | 백그라운드 실행 |
| `skills` | ❌ | — | 사전 로드할 스킬 |
| `mcpServers` | ❌ | — | 에이전트 전용 MCP 서버 |
| `hooks` | ❌ | — | 에이전트 전용 훅 |

## 7. Squad Agent 시스템 — 8개 에이전트 전체 지도

```
┌─── 코어 파이프라인 ────────────────────────────────────────────┐
│                                                                │
│  squad-plan → [구현] → squad-review → squad-qa → squad-gitops  │
│    opus                  opus  ◄──┐    sonnet      haiku       │
│    R+W                   R-only   │    R+Bash      R+Bash      │
│                                   │                            │
│                            squad-refactor                      │
│                              opus · R+W                        │
│                                                                │
├─── 온디맨드 ───────────────────────────────────────────────────┤
│  squad-debug (opus)   squad-docs (sonnet)   squad-audit (opus) │
│  R+Bash               R+W                   R-only             │
└────────────────────────────────────────────────────────────────┘
```

> 코어 5개는 자동 체이닝, 온디맨드 3개는 필요 시 호출

## 8. 8개 에이전트 — 책임·모델·도구

| 에이전트 | 역할 | 모델 | 도구 | 라우팅 근거 |
|---------|------|------|------|------------|
| `squad-review` | 코드 리뷰 | opus | Read, Bash, Glob, Grep | 보안·로직 = 깊은 추론 |
| `squad-plan` | 기획/와이어프레임 | opus | +Write, Edit | 설계·엣지케이스 |
| `squad-refactor` | 리팩토링 | opus | +Write, Edit | 안전한 구조 변환 |
| `squad-qa` | 테스트/QA | sonnet | Read, Bash, Glob, Grep | 실행·결과 정리 |
| `squad-debug` | 디버깅 | opus | Read, Bash, Glob, Grep | 근본 원인 분석 |
| `squad-docs` | 문서 작성 | sonnet | Read, Write, Edit | 코드→문서화 |
| `squad-gitops` | Git 자동화 | haiku | Read, Bash, Glob, Grep | 패턴 작업·비용 최적화 |
| `squad-audit` | 보안 감사 | opus | Read, Bash, Glob, Grep | 보안은 놓치면 안 됨 |

## 9. 도구 순서 표준 (Tool Ordering)

코드 리뷰 도구 시, 일관된 순서로 권한을 명시한다.

| 유형 | 도구 순서 | 적용 에이전트 |
|------|----------|--------------|
| Read-only | `Read, Bash, Glob, Grep` | review, audit, debug, gitops |
| Read + Bash | `Read, Bash, Glob, Grep` | qa |
| Write (Bash 없음) | `Read, Write, Edit, Glob, Grep` | docs |
| Write + Bash | `Read, Write, Edit, Bash, Glob, Grep` | plan, refactor |

> 도구 순서는 **하드 제약**이다. `tools:` 필드에 없는 도구는 호출 자체가 차단된다.

## 10. 에이전트 구현 — squad-review 예제

```markdown
---
name: squad-review
description: >
  Expert code review agent. Use PROACTIVELY after code changes, commits,
  or when user says "리뷰", "review", "코드 리뷰", "PR 리뷰", "코드 봐줘".
  Pipeline: after implementation. APPROVE → /squad-qa,
  REQUEST_CHANGES → /squad-refactor
tools: Read, Bash, Glob, Grep
model: opus
maxTurns: 15
---

You are a senior staff engineer conducting thorough code reviews.

## Review Process
1. Run `git diff HEAD~1` to identify modified files.
2. For each file, read the full file for context.
3. Classify findings: Critical / High / Medium / Low.

## Rules
- NEVER modify any files. Read-only.
- Focus on the DIFF, not the entire codebase.
```

---

## 11. 에이전트 구현 — squad-plan (Write 권한 보유)

```markdown
---
name: squad-plan
description: >
  Pre-development planning. Produces user stories, wireframes (SVG/HTML),
  and implementation plans.
  Pipeline: START → implement → /squad-review
tools: Read, Write, Edit, Bash, Glob, Grep
model: opus
maxTurns: 25
---

## Workflow
1. Read existing code and docs to understand current state.
2. Output deliverables in order:
   a. User Stories (US-001, US-002, ...)
   b. Wireframe → docs/wireframes/{feature}.svg
   c. Implementation Plan

## Boundaries
**Will:** Create user stories, wireframes, plans
**Will Not:** Implement features (→ user implements, then /squad-review)
                Modify existing source code
                Run tests (→ /squad-qa)
                Commit (→ /squad-gitops)
```

> Write 권한이 있는 에이전트는 **반드시 `## Boundaries` 섹션 필수**

## 12. 모델 라우팅 — 비용·정확도의 트레이드오프

| 작업 성격 | 모델 | 이유 |
|----------|------|------|
| 보안·로직 검토 | **opus** | 미스 비용이 큰 깊은 추론 |
| 아키텍처 설계 | **opus** | 엣지케이스·트레이드오프 고려 |
| 안전한 리팩토링 | **opus** | 의미 보존을 위한 깊은 이해 |
| 디버깅 | **opus** | 근본 원인은 표면 너머 |
| 보안 감사 | **opus** | 놓치면 프로덕션 사고 |
| 테스트 실행 | **sonnet** | 단순 실행·결과 정리 |
| 문서 생성 | **sonnet** | 코드→자연어 변환 |
| 커밋 메시지 | **haiku** | 패턴 작업, 비용 최적화 |

> **글로벌 오버라이드:** `export CLAUDE_CODE_SUBAGENT_MODEL=sonnet`

## 13. 핵심 설계 원칙

### ① 단일 책임 원칙(SRP)을 에이전트 레벨에 적용
- `squad-review`는 **찾기만**, `squad-refactor`는 **수정만**, `squad-qa`는 **테스트만**
- 한 에이전트가 모든 일을 다 하지 않는다

### ② 도구 권한 ≠ 프롬프트 지시
- "수정하지 말라"는 프롬프트로 지시하면 무시될 수 있다
- 도구 자체를 제거(Write 미부여)하면 호출이 차단된다

### ③ 파이프라인 컨텍스트
- 각 에이전트의 description에 `Pipeline:` 라인을 넣어 워크플로우 위치 명시
- Claude Code 자동 위임과 SubagentStop 훅 양쪽에 컨텍스트 제공

### ④ 페르소나 + 절차 + 출력 형식
- 시스템 프롬프트 = "당신은 X다" + "이 순서로 일하라" + "이 형식으로 보고하라"

## 14. 프로젝트별 오버라이드

전역 에이전트(`~/.claude/agents/`)와 프로젝트 에이전트(`.claude/agents/`)는 **이름이 같으면 프로젝트 버전이 우선**한다. 팀별 룰을 박아넣을 수 있다.

### 예시: `examples/project-override/squad-review.md`

```markdown
---
name: squad-review
description: >
  Expert code review for MyProject.
tools: Read, Bash, Glob, Grep
model: opus
maxTurns: 15
---

You are a senior staff engineer conducting reviews for MyProject.

## Project-Specific Rules

- TypeScript `any` is PROHIBITED — use proper types
- All API responses must use the `Result<T, E>` pattern
- Docker Compose labels must be strings
- Database queries must use parameterized statements
```

> 전역 에이전트의 일반 룰 + 프로젝트 룰을 **계층적**으로 운영. 팀 합류 시 `git clone`만 받으면 즉시 동일한 리뷰 기준 적용.
