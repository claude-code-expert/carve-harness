# 7.1.5 SubAgent CLI의 기능과 구현 II

## 슬래시 커맨드·훅 시스템·개발 주의사항

> 이전 장(7.1.4)에서는 에이전트 정의 형식과 8개 에이전트의 구조를 다뤘다.
> 이 장에서는 **에이전트를 호출하고 자동화하는 메커니즘**을 다룬다.

---

## 학습 목표

1. **슬래시 커맨드** — 사용자가 명시적으로 에이전트를 호출하는 방식
2. **3개의 훅 시스템** — UserPromptSubmit, SubagentStart, SubagentStop
3. **자연어 자동 라우팅** — Squad Router의 키워드 매칭 메커니즘
4. **파이프라인 체이닝** — 에이전트 완료 시 다음 단계 자동 안내
5. **개발 주의사항** — 안티패턴, 안전 규칙, 디버깅 방법

## 1. 슬래시 커맨드 정의

### 파일 구조 — `~/.claude/commands/squad-{name}.md`

```markdown
---
description: "Code review. Usage: /squad-review [scope]"
allowed-tools: Agent
---
Invoke the squad-review subagent. Default scope: last commit diff.
$ARGUMENTS
```

### 핵심 포인트

- `allowed-tools: Agent` — 슬래시 커맨드는 **Agent 도구만** 허용 (다른 도구 차단)
- `$ARGUMENTS` — 사용자가 입력한 인자가 그대로 치환됨
- 본문은 **Claude에게 주는 지시문** — "Invoke the squad-X subagent"라고 쓰면 Claude가 Agent 도구로 위임

## 3. 통합 커맨드 `/squad`

```markdown
---
description: "Invoke a Squad Agent. Usage: /squad <member> [task]"
allowed-tools: Agent
---

Parse the user's input:
- First word = member keyword (review, plan, refactor, qa, ...)
- Remaining text = task description

Available Squad Agents:
- squad-review: Code review
- squad-plan: Feature planning, user stories, wireframes
- squad-refactor: Refactoring (extract, simplify, rename, remove)
- squad-qa: Run tests and generate QA report
- ...

If no match, list all members and ask to choose.

$ARGUMENTS
```

> 첫 단어로 멤버를 선택하는 **수동 라우터** — `/squad review` == `/squad-review`

## 4. 훅 시스템 — 전체 구조

Claude Code의 훅은 특정 이벤트 발생 시 외부 셸 스크립트를 실행하는 메커니즘이다.

```
┌─── Squad Agent의 3개 훅 ───────────────────────────────────────┐
│                                                                │
│  1. UserPromptSubmit  → squad-router.sh                        │
│     사용자 프롬프트 제출 시 키워드 분석 → 자동 라우팅                   │
│                                                                │
│  2. SubagentStart     → subagent-chain.sh                      │
│     서브에이전트 시작 시 OS 알림 발생                                │
│                                                                │
│  3. SubagentStop      → subagent-chain.sh                      │
│     서브에이전트 종료 시 OS 알림 + 다음 단계 안내                      │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

> 모두 `install.sh`가 `~/.claude/settings.json`에 자동 등록한다.

---

## 5. settings.json 훅 등록 형식

```jsonc
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "matcher": "",
        "hooks": [{ "type": "command",
                    "command": "bash ~/.claude/hooks/squad-router.sh" }]
      }
    ],
    "SubagentStart": [
      {
        "matcher": "",
        "hooks": [{ "type": "command",
                    "command": "zsh ~/.claude/hooks/subagent-chain.sh" }]
      }
    ],
    "SubagentStop": [
      {
        "matcher": "",
        "hooks": [{ "type": "command",
                    "command": "zsh ~/.claude/hooks/subagent-chain.sh" }]
      }
    ]
  }
}
```

> `matcher: ""` — 모든 이벤트에 매칭. 특정 에이전트만 매칭하려면 `"squad-*"` 등 사용 가능.

## 6. 훅 ① — Squad Router (UserPromptSubmit)

### 동작 흐름

```
1. 사용자: "이 코드 보안 검사해줘"
   ↓
2. Claude Code가 훅 실행 (Claude에 프롬프트 전달 직전)
   stdin → {"prompt": "이 코드 보안 검사해줘"}
   ↓
3. squad-router.sh가 키워드 매칭
   "보안" 감지 → AGENT="squad-audit"
   ↓
4. stdout으로 JSON 출력 (context injection)
   {"hookSpecificOutput": {
     "hookEventName": "UserPromptSubmit",
     "additionalContext": "[Squad Router] You MUST delegate to squad-audit..."
   }}
   ↓
5. Claude가 system-reminder로 주입된 컨텍스트를 보고 위임
   → Agent(subagent_type="squad-audit", prompt="...")
```

> 핵심: `hookSpecificOutput.additionalContext`로 출력한 텍스트는 system-reminder로 주입됨. **프롬프트 자체는 변경되지 않는다.**

## 7. Squad Router 매칭 로직 — 3단계

```bash
# Phase 1: SKIP (스킵 조건)
case "$LOWER" in
  "/"*) exit 0 ;;                            # 슬래시 커맨드는 스킵
  *"--no-route"*|*"#direct"*) exit 0 ;;      # 옵트아웃 마커
esac
[ "$SQUAD_ROUTER" = "off" ] && exit 0        # 환경변수 글로벌 비활성화

# Phase 2: CONFLICT (다단어 패턴 우선 매칭)
case "$LOWER" in
  *"pr 리뷰"*) AGENT="squad-review" ;;       # PR 리뷰 → review
  *"pr 작성"*) AGENT="squad-gitops" ;;       # PR 작성 → gitops
  *"보안 리뷰"*) AGENT="squad-audit" ;;      # 보안 리뷰 → audit (review 아님)
  *"빌드 에러"*) AGENT="squad-debug" ;;      # 빌드 에러 → debug
esac

# Phase 3: GENERAL (단일 키워드, 우선순위 순)
# audit(1) > debug(2) > plan(3) > refactor(4)
# > docs(5) > gitops(6) > qa(7) > review(8)
```

## 8. Squad Router 키워드 매핑

**총 80개 키워드** (한국어 42 / 영어 38)가 8개 에이전트에 매핑된다.

| 우선순위 | 에이전트 | 주요 키워드 |
|---------|---------|------------|
| 1 | squad-audit | 보안, security, 취약점, vulnerability, OWASP, 시크릿, injection, xss, csrf |
| 2 | squad-debug | 디버그, debug, 에러, error, 버그, bug, 크래시, traceback, exception |
| 3 | squad-plan | 기획, 설계해, 와이어프레임, wireframe, 유저스토리, brainstorm, spec |
| 4 | squad-refactor | 리팩토링, refactor, 클린업, cleanup, 추출, extract, 중복 제거 |
| 5 | squad-docs | 문서화, 문서, document, readme, jsdoc, tsdoc, 주석, comment |
| 6 | squad-gitops | 커밋, commit, 체인지로그, changelog, 릴리즈 노트, conventional |
| 7 | squad-qa | 테스트, test, qa, 검증, 린트, lint, 타입체크, regression |
| 8 | squad-review | 리뷰, review, 코드 검토, 검토해, diff 봐 |

> **광범위한 키워드(review)는 가장 마지막 순위** — 다른 키워드가 먼저 매칭되도록.

## 9. Router 비활성화 옵션

| 방법 | 범위 | 예시 |
|------|------|------|
| `--no-route` 프롬프트 추가 | 건별 | `"리뷰해줘 --no-route"` |
| `#direct` 마커 | 건별 | `"리뷰해줘 #direct"` |
| `SQUAD_ROUTER=off` 환경변수 | 전역 | `export SQUAD_ROUTER=off` |
| `/squad-*` 슬래시 커맨드 | 자동 | 슬래시 커맨드는 자동 스킵 |

> 사용자가 명시적으로 슬래시를 입력했다면 자동 라우팅을 건너뛴다 — **명시적 의도 존중**

## 9-1. 키워드 선정 3대 기준

키워드를 무작정 늘리면 오탐(false positive)이 폭발한다. Squad Router는 다음 3가지 기준으로 80개 키워드를 선별했다.

### ① Hit rate (적중률)
"이 키워드가 들어간 프롬프트가 정말로 해당 에이전트를 의도하는가?"
- ✅ `보안` — 사실상 100% 보안 감사 의도
- ❌ `정리해` — "코드 정리"일 수도 있고 "회의록 정리"일 수도 있음 (제외)

### ② False Positive Resistance (오탐 저항성)
"일상 대화·코드 리뷰 외 맥락에서도 매칭될 가능성은?"
- ❌ `dry` — 3글자, "dry run" 등 일반어 (제외)
- ❌ `docs` — 파일 경로 `docs/` 매칭 (제외)
- ❌ `안돼` — 한국어 일상 대화에서 매우 빈번 (제외)

### ③ Conflict Safety (충돌 안전성)
"여러 에이전트에 걸치는 키워드는 다단어 패턴으로 분리한다."
- `PR 리뷰` → review / `PR 작성` → gitops
- `보안 리뷰` → audit (review 아님)

## 9-3. 제외된 키워드 — 왜 안 채택했나

| 키워드 | 매칭 후보 | 제외 사유 |
|--------|----------|----------|
| `docs` | squad-docs | 3글자, 경로 `docs/`에서 오탐 |
| `dry` | squad-refactor | 3글자, 일반 영단어 |
| `fail` / `실패` | squad-debug | "test failed"는 디버그 의도 아님 |
| `안돼` / `안되` | squad-debug | 한국어 일상 대화에서 너무 빈번 |
| `정리해` | squad-refactor | "회의록 정리" 등으로 오탐 |
| `verify` | squad-qa | 범용어, QA 특정 아님 |
| `pull request` | squad-gitops | "PR review"와 충돌 |
| `피알` / `풀리퀘` | squad-gitops | 슬랭 축약, FP 우려 |
| `코드 확인` / `코드 체크` | squad-review | 너무 범용 |
| `planning` | squad-plan | `plan ` (공백 포함)으로 커버 |
| `돌려` / `통과` | squad-qa | 일반 동사 |

> **교훈:** "더 많이 매칭"이 아니라 "정확히 매칭"이 좋은 라우터. 매칭 못 한 프롬프트는 사용자가 슬래시 커맨드로 보강하면 된다.

## 10. 훅 ② — SubagentStart / Stop (체이닝)

### subagent-chain.sh 핵심 로직

```bash
# stdin으로 들어오는 JSON에서 이벤트 종류와 에이전트 이름 추출
EVENT=$(jq -r '.hook_event_name' < "$TMPFILE")
AGENT_NAME=$(jq -r '.agent_type' < "$TMPFILE")

# squad-* 에이전트만 처리
case "$AGENT_NAME" in squad-*) ;; *) exit 0 ;; esac

if [ "$EVENT" = "SubagentStart" ]; then
  notify "🚀 Squad: ${DISPLAY_NAME}" "Status: RUNNING"
  play_sound "Pop"

elif [ "$EVENT" = "SubagentStop" ]; then
  case "$AGENT_NAME" in
    "squad-plan")     NEXT="→ implement, then /squad-review" ;;
    "squad-review")   NEXT="→ /squad-refactor or /squad-qa" ;;
    "squad-refactor") NEXT="→ /squad-review to verify" ;;
    "squad-qa")       NEXT="→ /squad-gitops commit" ;;
    ...
  esac
  notify "✅ Squad: ${DISPLAY_NAME}" "COMPLETED ${NEXT}"
fi
```

## 11. 크로스 플랫폼 알림

Claude Code는 TUI(터미널) 앱이라 훅의 stdout/stderr가 화면에 표시되지 않는다. **OS 네이티브 알림이 유일한 시각 채널**이다.

| 플랫폼 | 알림 방식 | 사운드 |
|--------|----------|--------|
| **macOS** | `osascript -e "display notification ..."` | `afplay` (Pop/Glass) |
| **Linux** | `notify-send` | `paplay` / `aplay` |
| **Windows/WSL** | `powershell.exe ... New-Object -ComObject Wscript.Shell` | — |

```bash
notify() {
  case "$(uname -s)" in
    Darwin)   osascript -e "display notification \"${body}\" with title \"${title}\"" ;;
    Linux)    command -v notify-send && notify-send "${title}" "${body}" ;;
    CYGWIN*|MINGW*|MSYS*) powershell.exe -Command "..." ;;
  esac
  # WSL 감지: Linux 커널이지만 Microsoft 호스트
  [ -f /proc/version ] && grep -qi microsoft /proc/version && powershell.exe ...
}
```

> 1. 에이전트 description의 `Pipeline:` 라인 (Claude 자동 위임용)
> 2. SubagentStop 훅의 NEXT 메시지 (사용자 안내용)

## 14. 실전 시나리오 — 자연어 입력

```
사용자: "이 코드 보안 검토 좀 해줘"
        ↓
[Squad Router] "보안" + "검토" 감지
        ↓
        Phase 2 충돌 검사: "보안 리뷰" 패턴 → squad-audit
        ↓
[Claude] system-reminder 수신: "Use squad-audit subagent"
        ↓
[Agent] subagent_type="squad-audit" 호출
        ↓
[OS 알림] 🚀 Squad: audit RUNNING
        ↓
[감사 실행] OWASP Top 10, 시크릿 검사, npm audit, ...
        ↓
[OS 알림] ✅ Squad: audit COMPLETED — Address findings before deploy.
        ↓
[메인] 보안 감사 보고서 (요약)만 메인 컨텍스트로 반환
```

## 14-1. 실전 시나리오 A — 신규 기능 개발 (8단계)

> 출처: `docs/SQUAD-COMMANDS-GUIDE.md` 시나리오 A

```bash
# 1단계: 기획
/squad-plan 비밀번호 재설정 (이메일 토큰 방식)
# → docs/wireframes/password-reset.svg
# → US-001 ~ US-004, 구현 8단계

# 2단계: 구현 (사용자 직접 코드 작성)

# 3단계: 리뷰

# → REQUEST_CHANGES: 토큰 만료 시간 하드코딩 발견

# 5단계: 재리뷰 → APPROVE / 6단계: /squad-qa → PASS

# 7단계: 커밋 + PR
/squad-gitops commit && /squad-gitops pr

# 8단계: 배포 전 보안 감사
/squad-audit src/auth/
```

## 14-2. 실전 시나리오 B — 프로덕션 버그 긴급 대응

## 15. 개발 주의사항 ① — Bash 화이트리스트/블랙리스트

Bash 도구를 가진 에이전트는 **반드시 명령어 제한 섹션**을 둔다.

```markdown
## Allowed Commands

```
git diff, git log, git show, git status
grep, cat, wc, find, head, tail
```

## NEVER Run

```
npm, rm, mv, git commit, git push
Any write or destructive operation
```
```

### 왜 필요한가
- Bash 도구는 강력하다 — 시스템 전체를 망가뜨릴 수 있다
- 화이트리스트는 **할 수 있는 것**, 블랙리스트는 **절대 안 되는 것**
- 두 섹션을 모두 두는 이유: LLM이 화이트리스트만 보고 우회를 시도할 수 있기 때문

---

## 16. 개발 주의사항 ② — Boundaries (Will / Will Not)

Write 권한이 있는 에이전트는 **반드시 `## Boundaries`** 섹션 필수.

```markdown
## Boundaries

**Will:**
- Read and analyze code changes for bugs, security issues, and quality
- Run read-only git and grep commands for context

**Will Not:**
- Modify any files (→ /squad-refactor)
- Run tests (→ /squad-qa)
- Commit or push changes (→ /squad-gitops)
```

### 핵심 패턴
- 각 "Will Not" 항목에 **위임 대상 명시** (`→ /squad-refactor`)
- 책임이 모호한 부분이 다른 에이전트로 자연스럽게 흘러감
- 에이전트가 자기 영역을 넘으려 할 때 자기 검열 신호 역할

---

## 17. 개발 주의사항 ③ — 안전 프로토콜

### squad-refactor의 사전 체크포인트 (필수)

```markdown
## Safety Protocol (MANDATORY — execute before ANY file modification)

1. Run `git stash list && git status` to verify clean working state.
2. Run `git stash push -m "pre-squad-refactor-checkpoint"`
   if there are unstaged changes.
3. Proceed with refactoring.
4. After all changes, run `git diff --stat` to summarize.

If `git status` shows uncommitted changes that aren't yours,
STOP and report.
```

### 핵심 원칙
- **Write 권한 = 자동 롤백 가능 상태 보장 의무**
- 에이전트가 작업 시작 전 stash로 체크포인트 생성
- 사용자의 in-progress 작업을 절대 덮어쓰지 않음

---

## 18. 개발 주의사항 ④ — 안티 패턴 (NEVER)

### squad-refactor에 명시된 금지 사항

```markdown
## Anti-Patterns (NEVER)

- Refactor and fix bugs simultaneously   # 리팩터와 버그 픽스를 섞지 말라
- Rename across the entire codebase      # 전체 코드베이스 일괄 리네임 금지
- Add abstractions "for the future"      # 미래 가정 추상화 금지
- Delete or modify test files            # 테스트 파일 손대지 말라
- Change public API without updating
  all callers                            # 공개 API 변경 시 모든 호출자 갱신
```

### 일반 원칙
1. **한 에이전트는 한 가지만** — review는 찾기만, refactor는 수정만
2. **scope 제한** — refactor는 module/directory 레벨까지만, cross-module은 STOP
3. **timeout 강제** — `timeout 120 npm test`로 hang 방지

## 20. 개발 주의사항 ⑥ — 셸 스크립트 견고성

```bash
#!/bin/bash
set -euo pipefail              # ① 에러 즉시 종료, 정의 안 된 변수 차단

TMPFILE=$(mktemp)              # ② 임시 파일 사용
trap 'rm -f "$TMPFILE"' EXIT   # ③ 종료 시 자동 정리

# ④ jq 가드 — jq 없으면 조용히 종료 (사용자 환경 다양성 대응)
if ! command -v jq &>/dev/null; then exit 0; fi
if ! jq empty < "$TMPFILE" 2>/dev/null; then exit 0; fi

# ⑤ glob 패턴 우선순위 — 긴 패턴이 먼저 와야 함
case "$LOWER" in
  *"문서화"*) AGENT="squad-docs" ;;   # 긴 패턴 먼저
  *"문서"*)   AGENT="squad-docs" ;;   # 짧은 패턴 나중
esac
# 반대로 쓰면 shellcheck SC2221/SC2222 — unreachable pattern 경고
```

## 23. 메타 케이스 스터디 — Squad Router는 squad-plan으로 만들어졌다

> **이 시스템의 가장 흥미로운 사실:** Squad Router 훅(v1.3.0)은 **`squad-plan` 에이전트가 직접 작성한 기획서** (`docs/plans/squad-router-hook.md`, 497줄)로 만들어진 산출물이다.

### 산출물 = 자기 참조 사례

```
[v1.2.x 시점]
  squad-plan 에이전트 (이미 존재)
        │
        │ /squad-plan "자연어 자동 라우팅 훅 만들어줘"
        ↓
[squad-plan이 산출]
  - US-001 ~ US-006 (6개 유저스토리)
  - Conflict Resolution Strategy
  - Phase 1/2/3 case 문 구조
  - Open Questions (Hook stdout 효과 검증 등)
        ↓
[사용자 구현]
  hooks/squad-router.sh + install.sh 수정
        ↓
[v1.3.0 릴리스]
  Squad Router 훅 — squad-plan이 만든 또 다른 squad 에이전트의 호출자
```

> "에이전트로 에이전트 시스템을 키운다" — Squad Agent가 자기 자신을 확장한 첫 번째 사례

## 24. 메타 케이스 스터디 — US-001~006 한눈에

| US | 제목 | 복잡도 |
|----|------|-------|
| **US-001** | 자연어 키워드 기반 자동 라우팅 | M |
| **US-002** | 충돌 키워드 우선순위 해결 (PR 리뷰 vs PR 작성 등) | M |
| **US-003** | 슬래시 커맨드와의 중복 방지 (`/`로 시작 시 스킵) | S |
| **US-004** | Opt-out — `--no-route`, `#direct`, `SQUAD_ROUTER=off` | S |
| **US-005** | install.sh 통합 (자동 등록, idempotent) | S |
| **US-006** | stdout 컨텍스트 주입 — Claude가 정확한 에이전트 위임 | S |

### squad-plan이 미리 식별한 Open Questions (5건)

| # | 미결 사항 | 위험도 |
|---|----------|-------|
| 1 | UserPromptSubmit stdout이 정말 Claude에게 컨텍스트로 전달되는가 | **HIGH** |
| 2 | "test" 키워드의 오탐 빈도 | MEDIUM |
| 3 | description의 PROACTIVELY와 라우터 이중 라우팅 | LOW |
| 4 | bash case 문 한글 인코딩 호환성 (macOS/Linux/WSL) | LOW |
| 5 | 다중 에이전트 매칭 — 현재 first-match-wins | LOW |

> **HIGH 위험도가 있다고 작업을 멈추지 않는다.** 위험을 명시적으로 기록하고 검증 단계에서 처리. 이것이 squad-plan이 생산하는 가치.

## 25. 메타 케이스 스터디 — Conflict Resolution 설계 의사결정

### squad-plan이 사전에 분석한 충돌 매트릭스

```
  case 문은 first-match-wins → 충돌 키워드는 일반 키워드보다 먼저 배치

  Phase 2 (CONFLICT — 복합 키워드):
  *"pr 리뷰"*       → squad-review   ("리뷰"가 의도)
  *"pr 작성"*       → squad-gitops   ("작성"이 의도)
  *"보안 리뷰"*     → squad-audit    ("보안"이 의도)
  *"빌드 에러"*     → squad-debug    ("에러"가 의도)
  *"빌드 확인"*     → squad-qa       ("확인"이 의도)
  *"코드 정리"*     → squad-refactor ("정리"가 의도)

  Phase 3 (GENERAL — 우선순위):
  audit(1) > debug(2) > plan(3) > refactor(4)
  > docs(5) > gitops(6) > qa(7) > review(8)
```

### 의사결정 근거

- **`audit`을 1순위로** — 보안 키워드 오탐 시 리스크가 가장 큼 (놓치면 프로덕션 사고)
- **`review`를 8순위로** — "리뷰/review"는 가장 범용적, 다른 키워드가 먼저 매칭되도록
- **단어 길이 ≠ 우선순위** — 의도 명확성이 더 중요

> **이 설계 자체가 squad-plan의 system prompt에서 학습된 사고 패턴.** "엣지케이스를 미리 식별하고 트레이드오프를 명시하라" — squad-plan에게 준 지시가 곧 산출물의 품질로 돌아온다.
